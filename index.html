The Premise

When you ask an LLM to calculate something, it doesn't actually compute. It predicts what the answer looks like.

That's why LLMs sometimes get math wrong. They're pattern-matching, not calculating. The usual workaround is to have the model write code, run it externally, and read the result back:

The weights are the CPU. Attention is the RAM.

Percepta's researchers built exactly this: a 7-layer transformer with compiled weights that executes WebAssembly programs — deterministically, step by step, for millions of steps.

But there's a problem they had to solve first.

The Bottleneck

In a transformer, every token looks at every previous token. This is self-attention, and its cost grows quadratically.

Attention cost

36 operations

For a program running 1 million steps, self-attention would need roughly 500 billion computations. Each new step has to scan the entire history.

To make a transformer-computer viable, they needed attention to be radically cheaper. Their solution is a beautiful geometric trick.

The Geometric Trick

Encode each memory position as a point on a parabola.

Instead of representing memory position j as a flat number, map it to a 2D point:

position j  →  point (2j, −j²)

This might seem arbitrary. But it unlocks something remarkable about convex hulls.

Why a Parabola?

Because every point on a parabola sits on the convex hull.

A convex hull is like stretching a rubber band around a set of points — the outer boundary. With random points, many end up trapped inside. But on a parabola, every single point is on the boundary.

Random Points

Parabola Points

This matters because finding the best memory match in attention is equivalent to finding a point on the convex hull. And if all points are on the hull, in sorted order, you can search it with a fast shortcut.

Attention as Geometry

In attention, the model computes a dot product between a query and every key. The highest dot product wins.

Geometrically, this is the same as asking: which point on the hull is furthest in the query direction?

Drag the query direction

The blue arrow is the query. The highlighted point has the highest dot product — the best memory match.

Since the parabola points are sorted along the hull boundary, we don't need to check every point. We can binary search.

The Transformer as Computer

Put it all together and the transformer can execute programs — step by step, deterministically.

Weights = CPU logic
Attention = RAM lookup
Tokens = machine state

The model reads a WebAssembly instruction, updates its internal state (registers, stack, memory), and produces the next state — all through its transformer layers. Each forward pass = one CPU cycle.

The actual model has just 7 layers, 18 attention heads (each 2-dimensional — the minimum needed for the convex hull trick), and about 964 compiled parameters. It runs at over 30,000 tokens/sec on a CPU.

What's Real, What's Open

This is genuinely novel work, but important caveats remain.

What's demonstrated

  • A transformer can deterministically execute programs via compiled weights
  • The convex hull trick gives O(log n) memory lookup in 2D attention
  • Execution is exact — same input always produces same output
  • 30k+ tokens/sec on CPU for this tiny, purpose-built model

Open questions

  • The weights are compiled, not learned — can this approach be trained via gradient descent?
  • The trick requires 2D attention heads — real LLMs use 64–128 dimensions per head
  • Can language understanding and program execution coexist in one model?
  • Is this actually faster than calling a normal WASM runtime?

The deepest implication: transformers may be capable of universal computation with logarithmic memory access. That would make them not just pattern matchers, but programmable compute substrates.

One-sentence takeaway

A transformer whose weights implement a CPU, and attention acts like the RAM lookup.
